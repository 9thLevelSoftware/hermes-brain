# Research: daem0n:learning-loop

## Summary

Daem0n-MCP's "learning loop" is three concentric rings. The inner ring — and the only place real learning happens — is dead simple: memories (SQLite rows, category=decision/pattern/warning/learning) carry `worked: bool|None` and `outcome: text`; `record_outcome()` sets them, versions the memory bi-temporally, updates Qdrant metadata, and a 1.5x FAILED_DECISION_BOOST multiplier in recall scoring makes failures louder in future retrieval. The middle ring is deterministic background processing: an IdleDreamScheduler fires after 60s of tool-call silence and runs four pluggable strategies (FailedDecisionReview, PendingOutcomeResolver, ConnectionDiscovery, CommunityRefresh) that re-query memory, count worked=True/False evidence, and persist "insights" as learning memories with excellent provenance tags. The outer ring is largely aspirational: a LangGraph Actor-Evaluator-Reflector Reflexion loop that is fully built (claims extraction, verification, code-gen/exec via E2B sandbox, reflection persistence, 5-occurrence consolidation) but never wired to any MCP tool or LLM — only its claim-verification submodule is actually exposed (via `reflect(action='verify')`/`verify_facts`). Crucially, the system contains no LLM client anywhere: every "cognitive" operation (debate, simulate, evolve) is deterministic scoring over recall() results, dressed in LLM-sounding language. The client agent (Claude) is the only intelligence; the server's honest contributions are storage, retrieval weighting, provenance, and workflow enforcement (the Sacred Covenant gating tools until briefing/context_check, plus git pre-commit blocking on stale pending decisions and worked=False files). The gold to keep: the worked/outcome field pair with retrieval boost, pending-outcome lifecycle pressure (pre-commit blocks + dream auto-resolution with dry-run default), dream provenance tagging, and rule staleness via outcome correlation. The cruft to drop: the unwired Reflexion graph, regex claim extraction, the evidence-counting "debate," and the theatrical framing.

## Subsystems

### Outcome recording (the real learning primitive)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\memory.py (record_outcome, ~line 1517)`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\memory.py (record_outcome tool, deprecated in favor of reflect(action='outcome'))`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\workflows.py (reflect dispatcher)`

Memory rows carry `worked: bool|None` and `outcome: str|None`. `record_outcome(memory_id, outcome, worked)` (1) sets both fields, (2) appends a MemoryVersion row with change_type='outcome_recorded' (bi-temporal audit trail), (3) re-upserts the Qdrant point with worked in metadata, (4) removes the id from SessionState.pending_decisions, (5) if worked=False, auto-adds the memory to ActiveContext at priority 10 and suggests the agent create an explicit warning memory, (6) clears the recall cache. Downstream consumption of the signal: recall scoring multiplies final_score by FAILED_DECISION_BOOST=1.5 for worked=False memories and prefixes them with '⚠️ This approach FAILED: {outcome}'; PreCommitChecker blocks commits touching files with worked=False memories and blocks when pending decisions are older than a configurable threshold (default 24h); dream strategies and cognitive tools count worked=True/False as evidence. Recording is the client agent's job — the server only nags (pre-commit warnings/blocks, briefing surfacing of pending decisions) and can auto-resolve via PendingOutcomeResolver.

**Assessment:** Genuinely excellent and the heart of the system. A tri-state worked flag + free-text outcome + retrieval boost + lifecycle pressure (pending-decision tracking, commit blocking) is a complete, minimal learning loop that requires no server-side LLM. The 1.5x boost is a crude but effective 'negative examples matter more' prior. Weakness: worked is binary (no partial success), the boost constant is unprincipled, and the whole loop depends on the agent remembering to call record_outcome — the enforcement ring exists precisely to compensate for that.

### Reflexion loop (Actor-Evaluator-Reflector, LangGraph)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\graph.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\nodes.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\state.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\claims.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\verification.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\code_gen.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\code_exec.py`

A real LangGraph StateGraph: START -> actor -> evaluator -> conditional(should_continue) -> reflector -> actor, with optional AsyncSqliteSaver checkpointing. Actor takes an optional llm_func (Callable[[str],str]); with none it emits a placeholder string '[Draft iteration N]...'. Evaluator: extract_claims(draft) via pure regex (three pattern families — MEMORY_PATTERNS like 'we decided...', OUTCOME_PATTERNS like 'that approach worked/failed', FACTUAL_PATTERNS like 'X returns Y'; OPINION_PATTERNS gate skips whole texts containing 'I think/probably/maybe'). verify_claims() embeds each claim (ModernBERT), recalls top-5 memories on the claim subject, computes cosine similarity; >=0.7 similarity = support unless has_negation_mismatch() (regex negation-pair matching from graph/contradiction.py) flags a conflict. Quality score = avg verification confidence - 0.2*conflicts - 0.05*unverified + 0.1 code bonus - 0.15 code-assertion-failure penalty. Loop exits at quality >= 0.8 or 3 iterations (warning logged at 2). Phase 14 adds code-augmented verification: is_code_verifiable() regex-selects numeric/date/code-behavior claims, generate_verification_code() (LLM or templates) emits Python asserts ending in print('ALL_ASSERTIONS_PASSED'), executed in an E2B sandbox with a 2-execution budget and a 6-way CodeFailureType classification (SUCCESS/ASSERTION_FAILURE penalize/boost quality; SYNTAX/IMPORT/TIMEOUT/SANDBOX errors are neutral). Reflector is trivial keyword mapping: 'CONFLICTS DETECTED' -> 'Remove or correct conflicting statements'. CRITICAL FINDING: run_reflexion/create_reflexion_app/persist_reflection are exported but never called by server.py or any MCP tool — the loop is dead code. No LLM client exists in the codebase; llm_func is always None in practice. Only the claims+verification submodule is live, exposed as verify_facts / reflect(action='verify'), where the CLIENT agent submits its own text for verification against stored memory.

**Assessment:** The architecture is textbook-correct Reflexion but it is theater in situ: without an LLM the actor produces placeholders, the reflector is string matching, and the whole graph is unwired anyway. The genuinely good part is claim verification as a standalone service — 'does this statement contradict stored memory?' via embedding similarity + negation detection is cheap, deterministic, and useful as a hallucination tripwire for memory-reference claims specifically. Regex claim extraction is brittle (English-only, misses paraphrase, FACTUAL_PATTERNS like 'X is Y' massively over-match) and the code-verification path is elaborate machinery (E2B sandbox, failure taxonomy) for a class of claims (arithmetic, stdlib behavior) that a frontier client model rarely gets wrong. Verdict: keep the concept 'verify memory-reference claims against the store before trusting them'; drop the graph, drop code-gen, drop regex extraction (the client LLM can extract claims itself).

### Reflection persistence & consolidation

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\persistence.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\consolidation.py`

persist_reflection() stores a reflection as a category='reflection' memory only if changed_behavior=True, tagged ['reflection','self-correction', error_type, 'sig:{16-hex sha256(error_type:normalized_content)}']. Dedup: if any reflection with the same sig exists, skip (a comment admits the occurrence counter was never implemented). create_reflection_from_evaluation() derives error_type from evaluator output (conflict > unverified_claim > factual_error > quality_improvement if delta > 0.1, else None). consolidate_reflections(): when >=5 reflections share a sig tag, create a permanent category='pattern' memory whose content is extract_common_elements() — literally a bag-of-words frequency count: words appearing in >=half the reflections, minus 13 stopwords, top 15 terms joined by spaces — and link it to sources via 'supersedes' edges. check_and_consolidate() scans up to 100 reflections, groups by sig, consolidates unconsolidated groups.

**Assessment:** The episodic->semantic consolidation IDEA (N similar corrections -> one permanent pattern with supersedes provenance) is the single best design in the reflexion package. The implementation is broken in two ways: (1) it is never invoked anywhere (dead code, like the rest of the loop), and (2) extract_common_elements produces word-salad, not a lesson — 'Learned pattern from 5 similar corrections: async sqlite session transaction commit error' is barely usable. Also the sha256 signature means only EXACTLY identical normalized content groups together, defeating the 'similar errors' intent; the 0.85 similarity_threshold parameter is accepted and ignored. In a successor: keep threshold-triggered consolidation with supersedes links, but have the client LLM write the pattern summary, and group by embedding similarity, not exact hash.

### Idle dreaming (scheduler + strategies)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\dreaming\scheduler.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\dreaming\strategies.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\dreaming\persistence.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\server.py (lines 230-345, wiring)`

IdleDreamScheduler: background asyncio task polls time.monotonic() at 1s granularity; after idle_timeout (settings.dream_idle_timeout, default 60s) of no tool calls it clears a user_active asyncio.Event and invokes the dream callback. CovenantMiddleware calls scheduler.notify_tool_call() on every MCP tool call, which resets the timer and sets user_active — strategies check user_active.is_set() before each unit of work and abort (session.interrupted=True), giving clean preemption. The callback (server.py) picks the most-recently-accessed ProjectContext and runs 4 strategies in sequence: (1) FailedDecisionReview — query worked=False, non-archived decisions older than min_age_hours, skip any reviewed within a cooldown window (detected by scanning learning memories tagged dream+re-evaluation+source-decision:{id}); for each, recall(decision.content[:200], limit=5), and classify: any evidence with worked=True -> 'revised'; <2 evidence items -> 'needs_more_data' (not persisted); else 'confirmed_failure'. (2) PendingOutcomeResolver — for decisions with outcome IS NULL: recall limit=10, count positive (worked=True) vs negative (worked=False or category=warning); decision tree: <2 total or <threshold directional -> insufficient_evidence; mixed -> flagged_for_review; unanimous positive/negative >= threshold -> auto_resolved_success/failure, which actually calls record_outcome() with '[DREAM AUTO-RESOLVED]' prefix — but ships dry_run=True by default, downgrading to flagged_for_review. (3) ConnectionDiscovery — group MemoryEntityRef rows from the last 168h by entity, find memory pairs sharing >=2 entities without an existing MemoryRelationship edge, create up to 20 'related_to' links at confidence 0.7, invalidate graph cache. (4) CommunityRefresh — if >=10 memories since last MemoryCommunity build, rerun Leiden community detection. Insights persist as category='learning' memories with tags [dream, re-evaluation, dream-session:{uuid12}, source-decision:{id}] and a context dict carrying evidence_ids, original content/outcome, timestamps; a session summary memory is written if any insights were generated.

**Assessment:** This is the best-engineered subsystem in the assigned area. The scheduler is clean (monotonic clock, cooperative preemption via Event, debounce, auto-start on race). Provenance is exemplary — every insight is traceable to its source decision, evidence set, and session. Cooldowns prevent re-review churn; dry-run-by-default on auto-resolution is exactly the right safety posture. The weakness is the analysis itself: 'evidence' is just recall hits counted by worked flag — FailedDecisionReview's 'revised' verdict means only 'some semantically nearby memory succeeded', which is a weak inference, and the insights it writes are formulaic sentences that then pollute recall results (each dream session adds learning memories that future recalls surface). ConnectionDiscovery and CommunityRefresh are honest graph maintenance, not learning, and are fine. Net: keep the scheduler and provenance model wholesale; replace the heuristic re-evaluation with either an actual LLM pass (successor has agent access) or drop the verdict and just SURFACE stale failures for the agent to reconsider.

### Cognitive tools (simulate_decision, evolve_rule, debate_internal)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\cognitive\simulate.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\cognitive\evolve.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\cognitive\debate.py`

All three are LLM-free deterministic queries. simulate_decision ('Temporal Scrying'): given a decision memory id, find its first MemoryVersion timestamp (fallback created_at), run recall(content[:200]) twice — once with as_of_time=decision_time (bi-temporal point-in-time query) and once now — then set-diff memory ids into new_evidence / invalidated_evidence / outcome_changes (worked flag flips among the overlap). confidence = min(1, total_changes/historical_count * 0.5). Output is a templated 'counterfactual assessment' paragraph. evolve_rule: staleness = 0.4*code_drift + 0.4*outcome_correlation + 0.2*age. code_drift = fraction of >3-char trigger tokens not found in the tree-sitter code index; outcome_correlation = failed/(worked+failed) among recall(trigger, limit=20) results, defaulting 0.5 with no data; age = min(1, days/365)*age_weight. Suggestions generated at staleness >= 0.2 ('update_trigger' per missing entity, 'add_warning' if correlation > 0.7, 'review_needed' if age_factor > 0.5); explicitly never auto-applies. debate_internal ('Adversarial Council'): both 'sides' call recall(f'{topic} {position}', limit=10); score_evidence = mean of relevance values with worked=True *1.5, worked=False *0.5, new-file diversity *1.1, normalized by len*1.5. The 'judge' compares two floats. Up to 5 'rounds' — but each round re-runs the identical queries, so scores can only change via the 5s recall cache expiring; convergence (delta < 0.05 both sides, min 2 rounds) is therefore near-guaranteed at round 2. Early exit with confidence 0 if both sides have <2 evidence memories. confidence = min(1, score_diff*2 + total_evidence/20). Consensus persisted as a learning memory with debate-id tags and all evidence ids.

**Assessment:** Honest ranking: simulate_decision is genuinely valuable — it is the one place the bi-temporal versioning pays off, answering 'what has changed since I decided this?' with a real, precise diff; keep the mechanism, drop the mystical prose. evolve_rule is a good idea (rules should decay; outcome correlation against rule triggers is a clever grounding) with mediocre execution — token-level code drift is noisy, and the 0.5 no-data default means most rules sit at ~0.4 staleness ambiguously. debate_internal is theater: the multi-round loop is fake (identical queries each round), the judge is a float comparison, and the persisted 'consensus' memories add noise. Its only real function — 'compare evidence strength for two options from memory' — is a single recall+aggregate call that doesn't need debate framing.

### Rules engine & matching

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\rules.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\context_triggers.py`

Rule model: trigger (natural language), must_do[], must_not[], ask_first[], warnings[], priority, enabled. check_rules(action): in-memory TF-IDF index over enabled rule triggers (rebuilt if DB changed since build, invalidated on rule mutation, results cached 5s); search(action, top_k=10, threshold=0.15); matched rules sorted by (priority, score); their four guidance lists concatenated and order-preserving-deduped; has_blockers = any must_not or warnings, producing an advisory message ('⚠️ STOP: Review warnings...'). add_warning_to_rule() lets learned failures append to a rule's warnings. ContextTriggerManager is a separate auto-recall system: triggers of type file_pattern (custom recursive **-glob via fnmatch), tag_match / entity_match (regex over lists); check_triggers matches current context and bumps trigger_count/last_triggered; get_triggered_context() then runs recall(recall_topic) per matched trigger, deduped by topic. Used to inject relevant memories when the agent touches matching files/tags/entities.

**Assessment:** Solid and appropriately simple. TF-IDF over rule triggers is the right weight class — rules are few, triggers are short, semantic embedding would be overkill; the 0.15 threshold is permissive by design since guidance is advisory. The must_do/must_not/ask_first/warnings quad is an excellent rule schema: it maps directly to how an agent should consume guidance. Critical honesty: check_rules is only enforcement-adjacent — nothing forces the agent to call it or obey it; actual enforcement lives in the covenant middleware and git hooks. ContextTriggers are a nice deterministic attention mechanism but require manual authoring; a successor could auto-derive them (e.g., from file_path on failure memories). Keep both, roughly as-is.

### Enforcement (sessions, pre-commit) & the Sacred Covenant

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\enforcement.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\covenant.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\hooks.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\transforms\covenant.py (CovenantMiddleware, not fully read)`

SessionManager: SessionState rows keyed by md5(project_path)[:8] + hour bucket (sessions expire hourly by construction). Tracks briefed (get_briefing called), context_checks (last 20, timestamped), pending_decisions (ids awaiting outcomes). Covenant flow: COMMUNE (get_briefing) -> SEEK COUNSEL (context_check) -> INSCRIBE (remember) -> SEAL (record_outcome). covenant.py classifies tools into three sets: exempt {get_briefing, health, context_check}; COMMUNION_REQUIRED (~30 tools, all reads and writes); COUNSEL_REQUIRED (9 mutating tools: remember, add_rule, prune, import, ingest...). CovenantEnforcer returns structured violation dicts (COMMUNION_REQUIRED / COUNSEL_REQUIRED / COUNSEL_EXPIRED with a 'remedy' field telling the agent exactly which tool call fixes it) instead of executing the tool; counsel has a 300s TTL. Enforcement is applied by FastMCP middleware (the decorators in covenant.py are deprecated). A PreflightToken (HMAC-SHA256-signed action+session+expiry, secret from DAEM0NMCP_TOKEN_SECRET with a hardcoded default) exists for cryptographic proof-of-consultation. PreCommitChecker (wired via installed git pre-commit hook -> CLI): blocks commit if any pending decision is older than 24h (configurable) or any staged file has a worked=False memory; warns on recent pending decisions and files with warning memories.

**Assessment:** The covenant is the most interesting behavioral idea: since the server cannot make the agent learn, it makes the desired workflow (read context before writing, record outcomes before committing) the path of least resistance by refusing tool calls with machine-readable remediation instructions. That works — agents follow structured 'remedy' hints reliably. The hour-bucketed session id is a hack (state silently resets on the hour boundary, mid-task). The PreflightToken is over-engineered: HMAC-signing a token that the same process issues and verifies, with a default secret, protects against nothing; session-state timestamps already do the job. Blanket COMMUNION_REQUIRED on all reads adds a mandatory get_briefing round-trip that costs tokens every session. Pre-commit blocking on stale pending decisions is the sharpest single enforcement lever in the whole system — it converts 'please record outcomes' into a hard gate at exactly the moment the agent knows the outcome. The theological naming (communion/counsel/inscribe/seal) is charming but leaks into API responses and wastes tokens.

## What Worked

- The worked/outcome pair on memories plus the 1.5x recall boost for failures — a complete learning signal with zero LLM cost; failures automatically become louder warnings in retrieval.
- Lifecycle pressure on outcome recording: pending_decisions tracked per session, surfaced in briefings, and hard-blocking git commits when >24h stale — this is what actually made the agent close the loop.
- IdleDreamScheduler's cooperative preemption design: monotonic-clock idle detection, asyncio.Event user_active checked between work units, clean interruption when the user returns.
- Dream provenance discipline: every autonomous insight carries dream-session id, source-decision id, evidence ids, and timestamps in both tags and context — fully auditable, and cooldown tags prevent re-review churn.
- PendingOutcomeResolver's conservative decision tree (unanimous evidence above threshold, mixed -> human review, dry_run=True default) — the right template for any autonomous write-back.
- Covenant violations as structured refusals with an explicit 'remedy' field (exact tool + args to call next) — steering the agent via machine-readable errors works far better than prose exhortation.
- simulate_decision's bi-temporal diff: 'what did I know then vs now' with new/invalidated/outcome-flipped evidence sets — the one feature that justifies memory versioning.
- Rule schema of must_do/must_not/ask_first/warnings matched via lightweight TF-IDF — right-sized, and add_warning_to_rule lets outcomes accrete onto rules.
- The consolidation CONCEPT: N similar episodic reflections -> one permanent pattern memory with supersedes links (even though the implementation was never wired and summarizes via word frequency).

## Weaknesses

- The entire Reflexion LangGraph (graph.py, nodes.py, persistence.py, consolidation.py, code_gen.py, code_exec.py) is dead code: run_reflexion/persist_reflection/check_and_consolidate are never called from server.py or any tool, and there is no LLM client, so the Actor emits placeholder strings even in tests.
- No server-side LLM anywhere means every 'cognitive' component is deterministic heuristics wearing LLM costumes — debate_internal's judge is a float comparison, its multi-round loop re-runs identical queries, and its persisted 'consensus' memories are noise.
- Regex claim extraction (MEMORY/OUTCOME/FACTUAL/OPINION pattern lists) is brittle: English-only, whole-text opinion veto discards mixed texts, and 'X is Y' patterns over-match wildly.
- Dream 'insights' are formulaic template sentences persisted as learning memories, which then surface in future recalls — the background loop slowly pollutes the store with low-information content.
- FailedDecisionReview's 'revised' verdict means only 'a semantically nearby memory has worked=True' — a weak inference presented as a re-evaluation.
- Consolidation dedup uses sha256 of exact normalized content (only identical reflections group), and pattern content is a bag-of-words dump; the accepted similarity_threshold param is ignored.
- Hour-bucketed session ids silently reset covenant state mid-task at every hour boundary; PreflightToken HMAC-signs tokens the same process issues and verifies, with a hardcoded default secret — security theater.
- worked is strictly binary; no partial outcomes, no confidence, no way to represent 'worked in context X but not Y'.
- surprise_score has schema, config, and calculator but no apparent call site in the write path — orphaned feature.
- Heavy per-project scoping (project_path threaded through everything, dream picks one most-recent context) conflicts directly with Hermes-Brain's agent-wide global memory goal.
- Mystical vocabulary (Sacred Covenant, Temporal Scrying, Adversarial Council, 'the daemon finds no rule inscribed') leaks into API responses and error messages, costing tokens and obscuring semantics.

## Reusable Ideas

- Keep the learning core exactly: memories with worked(bool|null)+outcome(text), record_outcome versioning, and a failure boost in retrieval scoring — but consider a graded outcome (worked/partial/failed + optional confidence) and a principled boost.
- Keep outcome-lifecycle enforcement: track pending decisions, surface them at session start, and gate commits (or task completion in Hermes) on resolving stale ones — this single mechanism drives the whole flywheel.
- Port the IdleDreamScheduler pattern (idle detection + pluggable strategies + user_active preemption + session provenance + per-strategy cooldowns) as the successor's background consolidation engine; since Hermes has agent/LLM access, replace heuristic verdicts with actual LLM re-evaluation of failed decisions during idle time.
- Adopt PendingOutcomeResolver's safety template for any autonomous memory mutation: evidence threshold, unanimity requirement, mixed-evidence -> flag for human, dry-run default, tagged audit memory for every action.
- Rebuild consolidation properly: cluster reflections/failures by embedding similarity (not exact hash), trigger at N occurrences, have the LLM write the distilled pattern, link sources via supersedes, and mark the pattern permanent — this is episodic->semantic memory done right.
- Keep claim verification as a standalone 'memory tripwire' service: when agent output references past decisions/outcomes, embed the claim, recall, check similarity >= 0.7 plus negation-mismatch for contradictions, and return verified/unverified/conflict markers — skip regex extraction (let the client LLM supply claims) and skip sandbox code verification.
- Keep simulate_decision's bi-temporal knowledge diff (then-vs-now recall with new/invalidated/outcome-flipped sets) as the payoff for memory versioning; it is the right primitive for 'should I revisit this decision?'.
- Keep rule staleness scoring but re-weight: outcome correlation (failures among memories matching the rule trigger) is the strong signal; token-level code drift is noise — replace with embedding similarity against current code index or drop it.
- Keep structured refusal-with-remedy as the enforcement mechanism (block tool, return exact next call), but scope it narrowly: gate only mutating operations, use real session lifetimes instead of hour buckets, and delete the HMAC token machinery.
- Wire surprise (kNN embedding distance at write time) into write-path triage: high-surprise memories resist decay and get consolidation priority; low-surprise duplicates merge — the code already exists, it just needs a call site.
- Drop entirely: the LangGraph actor/evaluator/reflector loop (the client agent IS the actor), debate_internal, code_gen/code_exec sandbox verification, PreflightToken, and the theological naming layer.
