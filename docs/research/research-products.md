# Research: research:products

## Summary

Production memory systems in 2025-2026 converged on a common playbook that differs sharply from academic RAG-heavy designs. The winners (ChatGPT, Claude, Claude Code, Gemini, OpenClaw) all use: (1) a small ALWAYS-INJECTED profile/index layer (stored facts, editable memory summary, MEMORY.md first-200-lines) rather than per-turn vector retrieval; (2) pre-computed summaries generated offline/asynchronously (ChatGPT's ~15-40 recent-conversation summaries, Honcho's deriver, Letta's sleep-time agents) instead of runtime search; (3) agent-driven file-based memory (Anthropic memory tool, Claude Code auto memory, OpenClaw markdown) where the model itself reads/writes plain files with progressive disclosure; (4) skills (agentskills.io SKILL.md) as the procedural-memory layer — 'memory of how' separate from 'memory of what'; and (5) strict prompt-cache discipline: never mutate the cached prefix mid-conversation, inject dynamic memory at the message tail. Hard lessons: reverse-engineering shows ChatGPT does NOT do RAG over chat history (it injects a profile + summaries and trims current-session first under pressure); memory is the #1 persistent prompt-injection surface (SpAIware, ZombieAgent); self-reported memory benchmarks are unreliable (Mem0-vs-Zep LoCoMo dispute with 58% vs 84% claims for the same system); and Hermes-Agent already has an open issue (#13631) documenting that Honcho's cadence-based system-prompt refresh breaks KV-cache on every prefix-caching backend — the fix pattern (byte-stable cached prefix + per-turn ephemeral injection) is exactly what Claude Code does with system-reminder tags appended to user messages.

## Findings

### ChatGPT memory + reference chat history (reverse-engineered architecture)

OpenAI's two-layer consumer memory: explicit 'saved memories' (the bio tool writing timestamped entries into a 'Model Set Context' block) plus 'reference chat history' (launched April 2025) which is NOT retrieval — it is a continuously maintained profile injected into every system prompt.

**Key techniques:**
- Six injected system-prompt sections: Model Set Context (timestamped facts), Assistant Response Preferences (behavioral guidelines with confidence scores), Notable Past Conversation Topics, Helpful User Insights, Recent Conversation Content (~15-40 timestamped summaries of USER messages only, no assistant replies), User Interaction Metadata (device/usage telemetry, intent_tags classification)
- No RAG/vector search over history — pre-computed summaries injected directly; 'sacrificing depth for speed'
- Token-pressure priority: current-session messages get trimmed FIRST; permanent facts and recent summaries always stay
- Automatic capture with confidence scoring and conversation intent classification; ~33 stored facts per typical user
- Hard-learned limitation: users cannot inspect/edit the auto-derived profile sections (only saved memories), creating trust/control gaps

**Evidence:** Reverse-engineered independently by Johann Rehberger (embracethered) and llmrefs by extracting the injected sections from live accounts; consistent with OpenAI's own Memory FAQ. Not officially documented by OpenAI, so section names/counts may drift; the two writeups disagree on summary count (~15 vs ~40), showing account-dependent variation.

**Applicability:** Directly validates a 'profile-first, retrieval-second' design for Hermes-Brain: maintain a compact always-injected user/agent profile block (facts + response preferences + rolling summaries of last N conversations) generated by the idle-time dreaming pipeline, and reserve the existing BM25+vector+RRF stack for on-demand deep recall rather than per-turn injection. The 'user messages only' summarization trick cuts summary cost/size roughly in half. Also adopt the trim-priority rule: under context pressure, drop session history before profile.

### Anthropic memory tool + context editing + compaction (official API)

Anthropic's GA memory tool (memory_20250818): a client-side file-based memory the model itself operates via six commands (view/create/str_replace/insert/delete/rename) against a /memories directory your app maps to real storage; paired with server-side context editing (clear_tool_uses_20250919, clear_thinking_20251015 under beta context-management-2025-06-27) and server-side compaction.

**Key techniques:**
- Model-driven memory: the API auto-injects a 'MEMORY PROTOCOL: always view memory directory first; assume interruption; record progress' system instruction — capture happens without user commands because the model is trained to check/write memory
- Client executes all ops — storage backend is yours (per-user dirs, DB keys); eligible for zero-data-retention
- Context editing clears old tool results server-side AFTER cache lookup, so cache prefixes survive; default trigger 60% of window, keep last N tool uses
- Memory + context editing benchmark claim: 84% token savings on long-running tasks (Anthropic's own number)
- Multisession pattern: initializer session creates progress log + feature checklist; every later session opens by reading them; mark done only after verification
- Security guidance: path-traversal validation mandatory, cap file sizes, expire files not accessed in a long time (their sanctioned forgetting mechanism)

**Evidence:** Official Anthropic platform docs (GA on Messages API, SDK helpers in 7 languages); 84% token-savings figure is Anthropic self-reported and task-dependent. Third-party writeups (Leonie Monigatti, others) confirm real-world behavior matches docs.

**Applicability:** Hermes-Brain should expose a memory-tool-shaped interface (the six commands) to the agent regardless of provider — it works model-agnostically as a normal tool on non-Anthropic models, and gets native training benefits on Claude. The /memories directory becomes a view over Hermes-Brain's store. hermes-agent issue #526 already requests the context-editing integration; the 'assume interruption, write progress to memory' protocol is the proven way to get automatic capture without user commands.

### Claude Code memory: CLAUDE.md hierarchy + auto memory (MEMORY.md index pattern)

The most battle-tested production pattern for agent self-memory: human-authored CLAUDE.md instruction files (managed-policy > user > project > local hierarchy, path-scoped .claude/rules/, @imports) plus 'auto memory' — Claude writes its own notes to ~/.claude/projects/<project>/memory/ with MEMORY.md as a size-capped index over on-demand topic files.

**Key techniques:**
- Two-tier auto memory: MEMORY.md index (only first 200 lines / 25KB loads every session) + unlimited topic files (debugging.md, api-conventions.md) read on demand — bounded always-loaded context, unbounded total memory
- Self-hygiene enforcement: after each write the harness measures MEMORY.md against the load limit and returns an ERROR forcing Claude to rewrite/condense the index — forgetting is forced by a hard budget, not left to model judgment
- Selective capture: Claude decides per-session whether anything is worth remembering (corrections, build commands, discovered preferences); 'remember X' user phrases route to auto memory automatically
- Separation doctrine: rules humans want enforced → CLAUDE.md; learned patterns that may change → auto memory; multi-step procedures → skills; hard guarantees → hooks (context is guidance, not enforcement)
- Cache-safe injection: CLAUDE.md is delivered as a user message after the system prompt; state updates go into appended reminder tags on the next user message so the cached prefix never mutates
- Subagent memory (v2.1.33, Feb 2026): each named subagent gets its own persistent store scoped to user or project
- Compaction survival: project-root CLAUDE.md is re-read from disk and re-injected after /compact

**Evidence:** Official Claude Code docs; used daily by a very large user base; Anthropic states hundreds of skills + these memory mechanisms are in active internal use. The 200-line/25KB numbers and error-on-overflow behavior are documented, versioned changelog items (v2.1.210-211).

**Applicability:** The MEMORY.md-index-plus-topic-files pattern with a hard load budget and write-time overflow errors is the single most directly portable design for Hermes-Brain's 'core memory': it gives automatic capture, automatic forgetting pressure (condense-or-lose), full human auditability (plain markdown, /memory command equivalent), and per-platform cheapness (works on a $5 VPS, no vector DB needed for the hot path). Per-subagent memory maps to Hermes skill/persona-scoped memory.

### Claude consumer apps memory (editable memory summary, project scoping, incognito)

Anthropic's consumer memory rollout (Aug 2025 explicit recall → Sept 2025 automatic synthesis for Team/Enterprise → Oct 2025 Pro/Max → all users by early 2026): automatic background memory with a user-visible, directly editable 'memory summary', per-Project memory isolation, and incognito chats.

**Key techniques:**
- Editable memory summary: the entire derived memory is rendered as one human-readable document the user can edit directly or adjust by telling Claude in-chat — memory transparency as a first-class artifact, in deliberate contrast to ChatGPT's uninspectable profile sections
- Project-scoped memory: each Project has separate memory, preventing cross-context bleed (work vs personal)
- Incognito/temporary chats that never write to memory (matched by Gemini's 72-hour Temporary Chats)
- Opt-in gradual rollout with import from ChatGPT/other assistants

**Evidence:** Official Anthropic blog + broad press coverage (VentureBeat, MacRumors); feature timeline corroborated across independent outlets.

**Applicability:** For a multi-platform assistant, the UX lessons are: (1) render the whole derived memory as an editable summary reachable from any platform (a /memory command in Telegram/Discord/CLI that dumps and accepts edits); (2) scope memory by context (per-project/per-group-chat) with a global user layer; (3) offer a no-memory mode per conversation. These three controls are now table stakes across all three major assistants.

### Gemini personalization ('personal context' from past chats)

**Key techniques:**
- Default-on implicit personalization from conversation history (profile-style, like ChatGPT's reference chat history)

- Single toggle granularity (less granular than OpenAI/Anthropic controls) — criticized for it

**Evidence:** Official Google blog + support docs; no technical architecture disclosed, so treat implementation details as unknown.

**Applicability:** Minor architecturally (Google publishes no internals), but the 72-hour bounded-retention 'temporary' mode is a good concrete spec for Hermes-Brain's ephemeral tier: messages in temp mode live in working memory only, with a TTL, and are provably excluded from the deriver/dreaming pipeline.

### agentskills.io Agent Skills standard + skills-as-procedural-memory

The open SKILL.md standard (originated by Anthropic, adopted by 26+ tools including Codex CLI, Gemini CLI, Copilot, Cursor — and Hermes): a skill is a directory with SKILL.md (YAML frontmatter: required name ≤64 chars kebab-case matching dir name, description ≤1024 chars; optional license, compatibility ≤500 chars, metadata map, experimental allowed-tools) plus optional scripts/, references/, assets/. Anthropic's engineering guidance explicitly frames skills as the procedural-memory layer agents can author for themselves.

**Key techniques:**
- Three-level progressive disclosure: metadata (~100 tokens, always loaded) → SKILL.md body (<5000 tokens, loaded on activation) → referenced files (on demand); keep SKILL.md under 500 lines, references one level deep
- Description field is the retrieval key — must say what AND when, with trigger keywords; misaligned metadata is the top failure mode
- Self-improvement loop: 'ask Claude to capture its successful approaches and common mistakes into reusable context and code within a skill'; when a skill underperforms, have the agent self-reflect and edit it — agents autonomously creating/refining skills is the stated trajectory
- Deterministic operations belong in bundled scripts, not tokens
- Security: skills are a dual attack surface (instructions + executable code); audit before install, treat as untrusted supply chain
- Validation tooling: skills-ref validate

**Evidence:** Official spec fetched in full; Anthropic reports hundreds of skills in daily internal use in Claude Code; 26+ tool adoption claim from ecosystem writeups (Firecrawl, inference.sh) — adoption breadth is credible, exact count is marketing.

**Applicability:** This is Hermes-Brain's procedural-learning substrate, already compatible by design. The concrete beyond-Daem0n move: close the loop where the Reflexion/dreaming pipeline PROMOTES repeated successful trajectories into new SKILL.md folders (draft → shadow-test → human-approve → active), and DEMOTES/edits skills whose outcome tracking shows failures. Memory of 'how' lives in versioned skill dirs (git-diffable), not in the fact store. The frontmatter metadata map can carry Hermes-Brain provenance (success count, last-used, derived-from-episode IDs) for staleness-driven pruning.

### Honcho by Plastic Labs (dialectic user modeling — currently integrated in Hermes)

AGPL-3.0 memory infrastructure: Postgres+pgvector storage layer (workspaces/peers/sessions/messages) plus an async 'deriver' worker that extracts facts, builds peer representations (global per-peer; optional local observer→observed representations), generates peer cards, short summaries every 20 messages / long every 60, and runs 'dreaming'. Queried via the Dialectic /chat endpoint (natural-language questions ABOUT a peer, 5 reasoning tiers minimal→max) and get_context (summaries + recent messages up to a token limit). Python/TS SDKs; managed service or self-host (needs Postgres+pgvector, an LLM API key, and two processes: API server + deriver worker).

**Key techniques:**
- Theory-of-mind peer modeling keyed by (observer, observed) pairs — same mechanism does self-representation and cross-peer models, which is genuinely differentiated for group chats (Discord servers, Telegram groups)
- Dialectic API: ask an expert about the user ('what is the user's mood today?') instead of retrieving raw facts — insight synthesis at query time
- Async deriver = capture without agent cooperation (messages just get logged; understanding accrues in background)
- get_context = prompt-ready context assembly under a token budget

**Evidence:** Official docs + repo (AGPL-3.0, ~two-service architecture confirmed); Plastic Labs self-claims 'Pareto frontier' on LongMemEval/LoCoMo — self-reported, unverified, and the Mem0/Zep dispute shows exactly these benchmarks are configurable to flatter anyone. The cache-invalidation problem is documented first-hand in the Hermes repo issue with measured miss rates.

**Applicability:** KEEP for what it is uniquely good at — cross-peer/group-chat user modeling and dialectic insight queries — but DEMOTE it from system-prompt injection to an on-demand tool + per-turn ephemeral block. Hermes-agent issue #13631 documents the concrete failure: Honcho's contextCadence/dialecticCadence refresh rebuilds cached system-prompt content, causing 100% KV-cache misses at cadence 1 (~25% at cadence 4) and multi-minute reprocessing of ~20K-token sessions on vLLM/llama.cpp/MLX/Anthropic/OpenAI. The issue's fix (remove Honcho from cached Layer-3 prompt, inject per-turn via _ext_prefetch_cache, enforce byte-stability) should be a Hermes-Brain design invariant. Also note: Honcho does NOT cover procedural/skill memory, outcome tracking, or forgetting policy — so it cannot be the whole brain; AGPL is fine for an open-source agent but matters for downstream embedders. Its deriver requires an extra worker process + LLM calls, a real cost on a $5 VPS — Hermes-Brain needs a degraded local mode when Honcho (or its budget) is absent.

### Prompt-cache-aware memory injection (the discipline every production harness converged on)

The design rules for injecting memory without destroying prefix caching (cached tokens ≈ 90% cheaper on Anthropic; prefix caching is also what makes vLLM/llama.cpp usable on long sessions): static content top, dynamic content bottom, and never mutate anything upstream of a cache breakpoint mid-conversation.

**Key techniques:**
- Order: tools → system → stable memory blocks → conversation; breakpoint on the last byte-identical block
- Never put timestamps, session IDs, cadence-refreshed memory, or shuffled tool lists in the prefix
- Claude Code's pattern: state/memory updates are appended as reminder tags to the NEXT USER MESSAGE, leaving the prefix untouched — memory injection without agent cooperation and without cache loss
- Server-side context editing (Anthropic) runs after cache lookup, so clearing old tool results doesn't invalidate the prefix — prefer it over client-side stripping when on Claude
- LangChain Deep Agents harness: explicit cache points so a memory update still yields a cache read on the unchanged subset
- Byte-stability invariant from hermes-agent #13631: cached prompt content changes only at deliberate compression events, never on a timer

**Evidence:** Anthropic official caching docs (90% cached-token discount), LangChain engineering blog, and empirical miss-rate measurements in hermes-agent #13631 (100% miss at cadence 1). Also an arXiv evaluation ('Don't Break the Cache', 2026) studying exactly this for long-horizon agents.

**Applicability:** This is the highest-leverage, lowest-effort win for Hermes-Brain and it is already half-specified in the repo's own issue tracker: define ONE injection contract — (a) stable profile/index block behind the cache breakpoint, updated only at session boundaries or explicit compaction; (b) all per-turn retrieval (RRF results, Honcho dialectic, freshness data) injected as ephemeral blocks in the latest user turn; (c) on Anthropic, use server-side context editing instead of client-side history surgery.

### Memory hygiene in production frameworks (Mem0 / Zep / Letta) + benchmark skepticism

The dedup/contradiction/staleness patterns the commercial memory layers actually ship, and the cautionary tale of their benchmark war.

**Key techniques:**
- Mem0 add-pipeline: extract durable facts from the conversation → compare against existing memories → LLM-adjudicated resolve (ADD / UPDATE / DELETE / NOOP) → store the resolved result; the documented production mistake is 'add every message', which creates near-duplicate facts and noisy retrieval within a week
- Zep/Graphiti bitemporal edges: superseded facts get invalidated (valid_from/valid_to) but preserved for audit — forgetting without data loss (Hermes already has bi-temporal KG; the production lesson is retrieval must FILTER on currency by default while keeping history queryable)
- TTL policies as the blunt-but-effective staleness control; flat vector stores fail at staleness because stale facts embed as close as current ones
- Letta sleep-time agents: a background agent SHARES the primary agent's memory blocks and rewrites them during idle time (compaction, archive management, learned-context distillation); primary can be a cheap fast model while the sleep-time agent uses a stronger model since it's not latency-bound
- Enterprise reality check: teams build governance on top (access control, audit logs, conflict resolution, retention policy) because no framework provides it out of the box
- Benchmark war: Zep claimed 84% on LoCoMo; Mem0's replication got 58.44%±0.20; Zep countered that Mem0 misconfigured them and claims 75.14%; contested variables (adversarial-question inclusion, API version, timestamp support) aren't pinned by the benchmark — treat ALL vendor memory-benchmark numbers, including Honcho's 'Pareto frontier' claim, as unverifiable marketing until reproduced in-house

**Evidence:** Mem0 pipeline from official docs/writeups; the dispute is fully public (Zep blog post, GitHub issue on getzep/zep-papers with corrected numbers, third-party analyses concluding neither result is independently reproducible). Letta sleep-time compute is shipped, documented product behavior.

**Applicability:** Hermes-Brain's dreaming loop should adopt the Mem0-style write-time adjudication (extract→compare→resolve) so the store converges instead of accumulating, layered on the existing bi-temporal contradiction detection; add default currency-filtered retrieval; run Letta-style asymmetric compute (cheap chat model, stronger idle-time model) for the dreamer. Build a small in-house eval from real Hermes transcripts rather than trusting LoCoMo/LongMemEval leaderboards.

### OpenClaw / memsearch: markdown-first memory for a multi-platform chat assistant

OpenClaw (ex-Clawdbot/Moltbot) — the closest production analog to Hermes (open-source, local-first, one agent across WhatsApp/Telegram/Discord/Slack/iMessage/Signal) — stores ALL memory as plain markdown (hand-curated MEMORY.md principles + auto-written daily log files) with a vector index built alongside; Milvus extracted the design into the open-source 'memsearch' library.

**Key techniques:**
- Files are the source of truth; the index is disposable — survives embedding-model swaps, is git-versionable, human-editable with any editor
- One shared workspace across all chat platforms: a preference set in Telegram applies in Discord — single brain, many mouths
- memsearch mechanics: watch (1500ms debounce re-index), chunk on heading boundaries, SHA-256 dedup to skip re-embedding, versioned chunk IDs hash(path:line:content:model_version) that auto-flag stale chunks on model upgrade, hybrid search 70% vector / 30% BM25, return top-3 chunks truncated to 200 chars (expand on demand)
- Compact workflow: LLM periodically distills old daily logs into durable principles (scheduled or manual) — forgetting-by-distillation with the originals still on disk (no regret: raw logs remain, only the loaded index shrinks)
- Deliberately omits temporal KGs and multi-persona modeling — transparency and portability over sophistication

**Evidence:** OpenClaw is the most-adopted open-source agent of 2025-2026 (massive GitHub traction, extensive third-party analysis); memsearch is extracted working code, not a paper. Milvus has an interest in selling vector DBs, but the described file layout matches OpenClaw's own docs (docs/concepts/memory.md).

**Applicability:** Confirms the cross-platform answer for Hermes: identity-keyed single workspace (user identity resolved across Telegram/Discord/CLI) over one memory store, with per-channel context notes rather than per-platform brains. The 'distill-don't-delete' compaction (raw episodic logs kept on disk forever, only the hot index is aggressively pruned) is the proven answer to 'forgetting without data loss regret'. The versioned-chunk-ID trick solves Hermes-Brain's embedding-upgrade staleness for ModernBERT/ONNX swaps.

### Memory as a persistent attack surface (SpAIware, ZombieAgent — what production learned the hard way)

Documented real-world exploit class: indirect prompt injection that writes MALICIOUS INSTRUCTIONS INTO long-term memory, where they persist across every future session — demonstrated against ChatGPT (SpAIware: continuous exfiltration of all future user messages via memory-implanted instructions appending text to attacker image URLs; Radware's ZombieAgent chaining connectors + memory).

**Key techniques:**
- Attack shape: untrusted content (web page, document, another chat member) → agent summarizes it → memory-write tool captures the injected 'remember to always...' instruction → payload executes in every future conversation until manually deleted
- Defense stack that emerged: treat all memory writes derived from untrusted content as tainted (provenance tagging), require review/quarantine for instruction-like memories, adversarial training, capability lockdown modes, transparent risk labeling
- Anthropic memory-tool docs mandate path-traversal validation (e.g. /memories/../../secrets.env), size caps, and expiry as baseline hardening
- Multi-platform agents are worse off: Telegram/Discord group messages mean OTHER USERS are an injection vector into a shared brain

**Evidence:** SpAIware is a published, vendor-acknowledged and patched exploit chain with PoC video (Rehberger); ZombieAgent covered by Dark Reading; Tenable published seven related ChatGPT vulns ('HackedGPT'). This is the best-documented failure mode of automatic memory in production.

**Applicability:** Hermes-Brain must have provenance on every memory record (who/what source, which platform, trust tier) and a rule that instruction-shaped memories (anything imperative that would change agent behavior) require elevated confirmation before entering the always-injected profile layer. In group chats, per-peer write isolation (Honcho's observer/observed model actually helps here) prevents user A from poisoning the agent's model of user B or its global behavior.

## Top Recommendations

- 1. Fix the injection contract first (cache-aware two-lane design): adopt a byte-stable always-injected layer (compact profile + MEMORY.md-style index behind the cache breakpoint, updated only at session boundaries/compaction) and route ALL dynamic memory — RRF retrieval results, Honcho dialectic answers, freshness data — into per-turn ephemeral blocks appended to the latest user message (Claude Code's reminder-tag pattern). This is already specified as the fix in hermes-agent issue #13631 with measured 100% cache-miss evidence; it is the cheapest, most proven win and every other feature depends on it.
- 2. Adopt the Claude Code auto-memory pattern as the core 'brain index': a size-capped MEMORY.md (200 lines / 25KB) that is the only always-loaded memory, backed by unlimited on-demand topic files, with write-time overflow errors forcing the agent to condense — this gives automatic capture, budget-forced forgetting, and plain-file human auditability that works identically on a $5 VPS and a GPU box.
- 3. Implement 'distill-don't-delete' forgetting (OpenClaw/memsearch): append-only raw episodic daily logs kept forever on cheap disk; the dreaming pipeline periodically distills them into principles and prunes only the HOT index. Combine with the existing bi-temporal KG so retrieval filters to current facts by default (Zep pattern) — forgetting without data-loss regret, reversible by re-indexing.
- 4. Close the procedural-learning loop through agentskills.io skills: the dreaming/Reflexion pipeline should PROMOTE repeated successful trajectories into draft SKILL.md folders (with provenance/success-count in frontmatter metadata) and demote/edit skills whose outcome tracking degrades — skills are the production-proven form of procedural memory (Anthropic runs hundreds internally) and Hermes is already spec-compatible.
- 5. Expose an Anthropic-memory-tool-shaped interface (view/create/str_replace/insert/delete/rename over /memories) as a normal tool on all providers, mapped onto Hermes-Brain storage — on Claude models you inherit trained memory behavior + the auto-injected 'assume interruption' protocol for free; on Anthropic also enable server-side context editing (issue #526) instead of client-side history surgery.
- 6. Keep Honcho but demote it: retain the deriver + dialectic API strictly for what nothing else does (cross-peer theory-of-mind in group chats, natural-language user-insight queries) as an on-demand tool and per-turn ephemeral context — never in the cached prefix — and build a degraded local mode (profile block + memsearch-style index) so memory works without the extra worker/LLM budget. Ignore its self-reported benchmark claims, as the Mem0-vs-Zep LoCoMo dispute proves these numbers are unreproducible; evaluate on your own Hermes transcripts.
- 7. Run Letta-style asymmetric sleep-time compute: the existing idle 'dreaming' loop should share the primary agent's memory blocks and use a STRONGER model than the chat path (it is not latency-bound) for fact adjudication (Mem0-style extract→compare→resolve on write), summary generation, and skill promotion.
- 8.
