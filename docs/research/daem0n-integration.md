# Research: daem0n:integration

## Summary

Daem0n-MCP's integration surface is a three-layer system: (1) an MCP tool surface consolidated from 67 individual tools into 8 verb-named workflow tools (commune/consult/inscribe/reflect/understand/govern/explore/maintain) plus 3 cognitive tools, each workflow being a thin FastMCP-registered shim that forwards a flat union of ~10-30 optional params to a per-workflow dispatch(action=...) module; (2) five Claude Code lifecycle hooks (SessionStart, PreToolUse Edit/Bash, PostToolUse Edit, Stop/SubagentStop) installed into ~/.claude/settings.json that make memory automatic by injecting briefing directives, blocking edits without a fresh 'preflight token' (5-min TTL context_check), enforcing must_not rules on bash, nudging inscribe() after significant edits, and regex-auto-capturing decisions from the transcript on Stop; and (3) a context-engineering stack (LLMLingua-2 compression with adaptive rates, JIT tiered thresholds, entity preservation, hierarchical community-summary strategies) that is mostly heavyweight machinery with only two real integration points. The true gold is the enforcement loop (hooks + covenant state making the agent USE its memory involuntarily), the 8-tool consolidation pattern, the auto-bootstrap briefing, and hooks that emit instructions to the model rather than doing work themselves. The cruft is the compression subsystem's ML dependency chain, dead modules (tool_search.py, prompt_templates.py), dual-import boilerplate, and per-project federation as a workaround for the absence of a global store.

## Subsystems

### 8-Workflow-Tool Consolidation (67 -> 8 + 3 cognitive)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\workflows.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\workflows\ (commune.py, consult.py, inscribe.py, reflect.py, understand.py, govern.py, explore.py, maintain.py, errors.py)`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\_deprecation.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\server.py`

Each of the 8 tools in tools/workflows.py is an @mcp.tool async function whose first param is `action: str` and whose remaining params are the FLAT UNION of every underlying action's params (consult has ~28 optional params; explore ~24). Dispatch: the tool lazily imports daem0nmcp/workflows/<name>.py and calls dispatch(action=..., **all_params). Each dispatch module has a VALID_ACTIONS frozenset and an if/elif chain routing to _do_X handlers; invalid actions raise InvalidActionError which surfaces as {error, recovery_hint} (recovery_hint lists valid actions). The handlers do NOT reimplement logic — they import the LEGACY tool functions from server.py (e.g. `from ..server import get_briefing`) and call them. Legacy tools are still decorated @mcp.tool at import time (server.py imports all tools/ modules, triggering registration) then server.py explicitly calls mcp.remove_tool() on a hardcoded list of ~60 deprecated names, so only 8 workflow + 3 cognitive tools are exposed to clients. A contextvar-based WorkflowCall context manager (_deprecation.py) suppresses the _deprecation payload/warning when legacy functions are invoked from inside a workflow dispatcher; if a legacy tool were called directly it would append a structured `_deprecation` dict to its result. Action counts: commune 6, consult 8, inscribe 9, reflect 3, understand 5, govern 6, explore 12, maintain 12 = ~60 actions. All tools default project_path from DAEM0NMCP_PROJECT_ROOT env and return a MISSING_PROJECT_PATH error dict with a human hint if absent.

**Assessment:** The consolidation was a genuine win for the stated goal — README claims 88% fewer tool definitions in context, and having 8 memorable verb-tools (commune=start, consult=before-acting, inscribe=write, reflect=close-loop) is a strong mental model the hooks can reference by name. BUT the implementation carries real costs: (1) flat param unions destroy per-action schema validation — a client can pass entity_name to consult(action='recall') and nothing complains; the MCP schema tells the LLM nothing about which params go with which action beyond docstring prose; (2) the register-then-remove-60-tools dance plus keeping all legacy @mcp.tool functions alive as the actual implementations is a migration artifact, not a design — double the code paths, deprecation shims, and dual try/except imports everywhere; (3) explore and maintain are grab-bag tools (12 actions each) where the verb no longer predicts the contents (maintain includes federation link_project). A successor should keep the small-verb-tool surface but design it natively: per-action typed sub-schemas (tagged unions) or one tool per verb with a documented action enum, and no legacy layer.

### Claude Code Lifecycle Hooks (the enforcement loop)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\session_start.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\pre_edit.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\pre_bash.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\post_edit.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\stop.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\_client.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\claude_hooks\install.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\enforcement.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\covenant.py`

install.py writes 6 entries into ~/.claude/settings.json hooks (PreToolUse matcher 'Edit|Write|NotebookEdit' and 'Bash', PostToolUse 'Edit|Write', Stop, SubagentStop, SessionStart), each running `"<sys.executable>" -m daem0nmcp.claude_hooks.<hook>`; it idempotently strips prior Daem0n/legacy entries first. _client.py is the shared kit: project detection via CLAUDE_PROJECT_DIR -> DAEM0NMCP_PROJECT_ROOT -> cwd, gated on `.daem0nmcp/` existing (hooks are silently no-ops in non-Daem0n projects); TOOL_INPUT env JSON parsing; block()=stderr+exit 2 (downgradable to warning via DAEM0N_HOOKS_PERMISSIVE=1); run_hook_safely() wraps main in a threading.Timer watchdog that os._exit(0)s on timeout (5s default, 10s pre-hooks, 15s stop) and converts ALL exceptions to exit 0 so hooks never break the user's workflow. MECHANICS PER HOOK: [session_start] does NOT compute the briefing — main() just prints '[Daem0n] IMPORTANT: Before responding to the user, call mcp__daem0nmcp__commune(action="briefing")...' and exits 0; the printed text lands in Claude's context as an instruction. (An unused _fast_briefing() raw-sqlite3 path exists that computes stats with busy_timeout=2000ms and upserts session_state.briefed=1 — superseded by the instruction-injection approach.) [pre_edit] spins up full SQLAlchemy managers, checks SessionManager.has_recent_context_check(project, max_age_seconds=300); if no context_check within 5 minutes it exits 2 with 'You must call consult(action=preflight, description=...) before editing' — hard-blocking the Edit. If the token is fresh, it recalls file-scoped memories + rules and prints them as '[Daem0n recalls for <file>]' with sections: Failed approaches (top 3, 150-char), Warnings, Rule warnings, Must do / Must NOT do — injected context right before the edit executes. [pre_bash] runs rules.check_rules(f'executing bash: {cmd[:200]}'); matching must_not => exit 2 block; warnings => printed guidance, exit 0. [post_edit] pure heuristic, no DB: if file extension is in a significant set (.py/.ts/.sql/...) AND change text contains any of ~24 patterns ('class ', '@mcp.tool', 'auth', 'migration', 'schema', ...) or len>500, prints '[Daem0n suggests] ... Consider: inscribe(action=remember, ...)'. [stop] reads CLAUDE_TRANSCRIPT_PATH JSONL, looks at last 5 assistant messages; fires only if COMPLETION_PATTERNS match ('all tasks complete', 'successfully implemented', 'pull request created'...) AND no EXPLORATION_PATTERNS ('let me explain', 'i found') AND no record_outcome already happened. Then regex-extracts up to 5 decisions via 6 DECISION_PATTERN regexes ('I'll use X', 'chose X because', 'warning: X', 'learned that X') with nearby-file-mention capture, writes them directly via MemoryManager.remember(rationale='Auto-captured from conversation'), and prints '[Daem0n auto-captured] N decision(s)... Remember to record_outcome()'. Anti-loop: per-session JSON state file in ~/.daem0nmcp/hook_state caps reminders at 2 within 2 turns. SERVER-SIDE COVENANT parallels this: ProjectContext carries briefed flag + context_checks list; CovenantMiddleware (FastMCP 3.0) blocks memory-writing tools until commune ran ('communion') and until a context_check within COUNSEL_TTL_SECONDS=300 ('counsel'); consult(action='preflight') = context_check, which records the timestamped check AND issues a serialized PreflightToken keyed to a session_id = md5(project)[:8] + hour bucket.

**Assessment:** This is the crown jewel and the direct answer to 'what made the agent actually USE its memory': memory usage was made INVOLUNTARY through three mutually reinforcing mechanisms. (1) Instruction injection at SessionStart — the hook doesn't fetch data, it commands the model to call commune(briefing) before doing anything, exploiting the fact that hook stdout becomes model context; this is more robust than computing the briefing in the hook because the MCP tool has the full stack (bootstrap, dreams, focus areas) and the model 'experiences' calling it. (2) Hard gating — pre_edit exit-2-blocks any Edit until consult(preflight) issued a token in the last 5 minutes, so the model literally cannot edit without first recalling relevant memories; the block message tells it the exact call to make (self-healing). Belt-and-suspenders: the same rule is enforced server-side via CovenantMiddleware for non-Claude-Code clients. (3) Closing the loop — post_edit and stop nudge inscribe/record_outcome with copy-pasteable call syntax. The engineering hygiene is also excellent: watchdog timers, exit-0-on-any-exception, permissive-mode env escape hatch, .daem0nmcp-dir gating, idempotent installer preserving foreign hooks. Weaknesses: pre_edit/pre_bash boot a full SQLAlchemy+aiosqlite stack per tool call (hence 10s timeouts) — the dead _fast_briefing shows the author knew raw sqlite3 was the right weight; the stop hook's regex decision extraction is crude and pollutes memory with low-quality 'Auto-captured from conversation' entries (server.py even has a maintain action 'purge_dream_spam' — a telltale that auto-generation outpaced curation); the 5-minute counsel TTL is arbitrary and can nag during long edits; post_edit keyword list ('auth','api','config') fires constantly on real codebases.

### Compression subsystem (LLMLingua-2 / adaptive / hierarchical / JIT / entity preservation)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\compression\compressor.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\compression\adaptive.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\compression\hierarchical.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\compression\jit.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\compression\entity_preserver.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\compression\config.py`

ContextCompressor lazy-loads microsoft/llmlingua-2-xlm-roberta-large-meetingbank (a token-classification model, torch+CUDA-auto) plus tiktoken cl100k_base for counting; only compresses above compression_threshold=4000 tokens; default rate 0.33 (3x); base_force_tokens preserve punctuation/brackets; drop_consecutive=True. CodeEntityPreserver regex-extracts up to 50 identifiers (13 patterns: def/class/function/const/interface/type/enum...) sorted by length ascending, plus ~60 hardcoded structural keywords, all fed as force_tokens; is_code_heavy = >5 code-indicator hits per 1000 chars. AdaptiveCompressor classifies content via the same density heuristic into CODE (rate 0.5), NARRATIVE (0.2), MIXED (0.33) and picks force-token strategy accordingly. HierarchicalContextManager routes by RecallPlanner query complexity: SIMPLE -> return Leiden community summaries verbatim (summaries ARE the compression), MEDIUM -> summaries + raw with moderate compression, COMPLEX -> raw with adaptive compression; takes skip_compression=True when JIT is the compression point (double-compression guard documented in a NOTE). JITCompressor is the actual production path: tiers soft>4K (rate 0.5), hard>8K (0.33), emergency>16K (0.2); dynamic rate = base_rate * sqrt(4000/token_count) clamped to [0.15, 0.6]; returns metadata {original_tokens, compressed_tokens, compression_rate, threshold_triggered, content_type}. Wire-in points: exactly two — memory.py:~1407 (recall results) and retrieval_router.py:~207 — plus a user-facing consult(action='compress') tool.

**Assessment:** Mostly dead weight for a successor. The layering is over-engineered: four classes (Compressor -> Adaptive -> Hierarchical -> JIT) where JIT admits in comments it supersedes Hierarchical's compression role, leaving Hierarchical as an assembly strategy with a skip flag — a classic accretion pattern. The cost side is severe: an XLM-RoBERTa-large model (~2GB, torch) lazy-loaded inside an MCP server just to shorten recall output, when the same system already has cheap levers (condensed=True recall mode, first-line-120-char summaries in briefings, community summaries, per-action limit params) that achieve the same token economy losslessly and are what the briefing actually uses. LLMLingua token-dropping also produces degraded, sometimes garbled text that a frontier model must then re-parse. The genuinely good ideas here are model-free: (a) tiered token thresholds with metadata telling the caller what happened (JIT's contract is clean), (b) 'community summaries ARE the compression' — pre-computed hierarchical summaries beat runtime compression, (c) content-type-aware conservatism for code, (d) sqrt-dampened dynamic rates as a pattern. A successor should keep the threshold/metadata contract and hierarchical summaries, and drop LLMLingua entirely in favor of structured truncation + summary tiers.

### Briefing / commune session-start experience

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\briefing.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\workflows\commune.py`

get_briefing (via commune action='briefing'): (1) get_statistics; (2) AUTO-BOOTSTRAP if total_memories==0 — runs 7 extractors and stores each as a tagged bootstrap memory: project identity from package.json/pyproject/Cargo/go.mod (name, description, top-10 deps), architecture (README first 2000 chars + top-level dir listing with child counts, noise dirs excluded), conventions (CONTRIBUTING + detected linter/formatter configs), CLAUDE.md/AGENTS.md instructions, git log last 30 commits, TODO/FIXME/HACK scan (limit 20, category='warning'), entry-point files 2 levels deep; (3) _fetch_recent_context — 5 most recent decisions/warnings each as {id, first-line summary truncated to 120 chars, worked}, 5 most recent FAILED memories (worked==False) with 60-char outcome, top-5 enabled rules by priority desc, plus total counts and a drill_down hint string ('Use recall(topic)... for full content'); (4) git changes since last memory timestamp (subprocess git); (5) optional focus_areas (max 4): condensed recall limit 3 each, top-3 items by relevance with 80-char summaries and a 'recall(area) for details' hint; (6) linked-projects summary (opens each linked DB for memory/warning counts); (7) recent dream sessions grouped by dream_session_id with insight summaries; (8) active context (pinned always-hot items, capped at 5, expired cleaned); (9) _build_briefing_message concatenates a one-line human message: totals, '[BOOTSTRAP] first run ingested: ...', '[WARNING] N failed approaches to avoid!', uncommitted file count, '[DREAM] daemon reviewed N decisions while idle...'. Sets ctx.briefed=True, unlocking the covenant. get_briefing_visual variant returns a UI resource hint for the MCP-Apps dashboard.

**Assessment:** This is the second-best subsystem. Prioritization philosophy is exactly right: failures first (failed approaches are surfaced in both the data and the alarm-toned message), then warnings, then decisions/rules; everything is id + first-line-truncated summary with explicit drill-down hints — the briefing is an index, not a dump, and it teaches the model how to get more. Auto-bootstrap is the standout idea: zero-memory cold start is converted into instant project awareness from artifacts that already exist (manifests, README, git, TODOs, CLAUDE.md), so the very first briefing is never empty and the agent's first impression of the memory system is that it already knows things. Weaknesses: the 1646-line module mixes bootstrap extractors, git subprocess calls, dream aggregation, and formatting; recency-only selection for decisions/warnings (no importance/decay scoring) means five trivial recent decisions can displace a critical old warning; linked-project summaries open every linked DB on every briefing (latency); bootstrap TODO scan as 'warning' category pollutes the warnings channel it later surfaces.

### Federation across projects (link_projects / LinkManager)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\federation.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\links.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\tools\briefing.py (_get_linked_projects_summary)`

Each project has its own isolated SQLite under <project>/.daem0nmcp/storage. link_projects(linked_path, relationship in {same-project, upstream, downstream, related}, label) stores a row in the SOURCE project's DB via LinkManager. Reads are federated: get_linked_db_managers() opens a live DatabaseManager per linked path (if its .daem0nmcp/storage exists) so recall(include_linked=True) and the briefing can query across projects; WRITES stay strictly local ('write isolation preserved' is the documented contract). consolidate_linked_databases(archive_sources) physically merges all linked memories into the current DB (monorepo transition path), optionally renaming source dirs to .daem0nmcp.archived. maintain workflow exposes link_project/unlink_project/list_projects/consolidate actions.

**Assessment:** Federation is a competent workaround for the fundamental architectural choice of per-project databases — and its existence is the strongest evidence FOR the successor's global-scope design. Lessons: (1) read-across/write-local is the right safety invariant and should survive as 'any agent can read cross-project knowledge, but provenance/write-scope stays attributed'; (2) the relationship taxonomy (upstream/downstream/related) is useful metadata a global store should keep as inter-project edges rather than DB links; (3) the pain points — opening N SQLite engines per briefing, links breaking when paths move, an entire consolidation tool needed just to restructure — all vanish with a single global store carrying a project/scope column; (4) manual linking never captures the common case (a lesson learned in project A that applies everywhere), which is precisely the continual-learning case Hermes-Brain targets. Verdict: don't port federation; replace it with scoped queries (global / project / file) over one store, and keep 'consolidate' only as a one-time importer from legacy per-project DBs.

### Config surface (pydantic-settings)

Single Settings(BaseSettings) with DAEM0NMCP_ env prefix + .env file, ~70 fields with pydantic Field validation (ge/le bounds): paths (project_root, storage_path, qdrant_path/url/api_key), context cache (max_project_contexts=10, context_ttl_seconds=3600), search tuning (hybrid_vector_weight=0.3, bm25_k1=1.5, bm25_b=0.75, rrf_k=60, diversity max_per_file=3), embeddings (nomic-ai/modernbert-embed-base, dim 256, onnx backend, query/document prefixes), recall limits per complexity (5/10/20), surprise scoring (k=5, boost 0.7), auto-zoom flags (disabled + shadow=True by default), ~15 dreaming knobs (idle 60s, cooldowns 72h/168h, dry_run=True default for auto-resolve), cognitive-tool params, TODO scanner excludes, file watcher (disabled by default). get_storage_path() implements resolution priority (explicit override -> project .daem0nmcp/storage -> centralized when running from the server's own dir) plus automatic legacy .devilmcp migration. **Assessment:** Solid, boring, correct: one typed settings object, env-prefixed, bounded fields, sane storage resolution with legacy migration baked in. Weaknesses: ~70 flat fields is a smell of feature accumulation (15 dream knobs nobody will tune); CompressionConfig being a second, env-invisible config system is inconsistent; no grouping/nesting so discoverability is poor. Successor: keep pydantic-settings + shadow-mode/dry-run convention, group into nested sub-models, and be ruthless about knob count.

### ProjectContext lifecycle / context_manager

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\context_manager.py`

ProjectContext dataclass bundles db_manager/memory_manager/rules_engine per normalized project path plus covenant state (briefed flag, context_checks list). Global dict cache with a custom RWLock: fast path read-lock lookup, slow path write-lock + per-project init lock with double/triple-checked locking; eviction is TTL (1h) + LRU (max 10 contexts) run opportunistically at most every 60s; per-asyncio-task active_requests refcounting (task done-callbacks release counts) prevents evicting a context mid-request; hold_context() async CM for long operations like Reflexion loops. Also hosts covenant check helpers (_check_covenant_communion/_counsel with COUNSEL_TTL_SECONDS=300) and _resolve_within_project path-traversal guard.

**Assessment:** Correct and carefully done (the two-phase eviction avoiding nested locks is textbook), but it is a large amount of concurrency machinery whose only reason to exist is the per-project-database decision — 10 cached DB engine stacks, LRU eviction, task-scoped refcounts. A global single-store successor deletes ~90% of this file. Worth keeping conceptually: covenant state living on the session/context object, path normalization as cache key, and the _resolve_within_project security guard.

## What Worked

- Hook-driven enforcement made memory automatic: SessionStart injects a direct instruction to call commune(briefing); pre_edit hard-blocks (exit 2) any Edit/Write until consult(preflight) ran within 5 minutes, with the block message containing the exact corrective call — the agent physically cannot skip recall before editing
- Hooks that emit instructions/suggestions to the model (with copy-pasteable tool-call syntax) instead of doing heavy work themselves — hook stdout becomes model context, and the model performs the memory operation through its normal tool path
- Defensive hook engineering: watchdog timer with os._exit(0), all exceptions -> exit 0, .daem0nmcp-dir gating so hooks are no-ops elsewhere, DAEM0N_HOOKS_PERMISSIVE downgrade, idempotent installer that preserves foreign hooks in ~/.claude/settings.json
- 8 verb-named workflow tools (commune/consult/inscribe/reflect/understand/govern/explore/maintain) collapsed 67 tool schemas to 8, cut context cost ~88%, and gave hooks/docs a stable vocabulary to reference ('call consult(action=preflight)')
- Auto-bootstrap on first briefing: 7 extractors (manifests, README+dir tree, linter configs, CLAUDE.md/AGENTS.md, git log 30, TODO scan, entry points) so a brand-new project's first briefing is never empty
- Briefing as an index, not a dump: id + first-line 120-char summaries, failures surfaced first with alarm framing ('N failed approaches to avoid!'), totals + explicit drill_down hints teaching the model how to fetch more
- Belt-and-suspenders enforcement: the same communion/counsel rules enforced both client-side (Claude Code hooks) and server-side (CovenantMiddleware + PreflightToken with 5-min TTL), so non-Claude clients are still disciplined
- Ship-inert convention for risky automation: auto_zoom shadow mode, dream_pending_dry_run=True — new autonomous behaviors log before they act
- JIT compression's caller contract: tiered thresholds (4K/8K/16K) with returned metadata (original/compressed tokens, tier triggered) so token management is observable
- Structured error recovery: InvalidActionError returns {error, recovery_hint} listing valid actions; MISSING_PROJECT_PATH error includes a how-to-fix hint — errors teach the model

## Weaknesses

- Flat param unions on workflow tools (consult has ~28 optional params) — no per-action schema validation, and the MCP schema can't tell the model which params belong to which action
- Migration-artifact architecture: all 60+ legacy tools still register via @mcp.tool at import then get mcp.remove_tool()'d; workflow dispatchers call legacy functions through server.py re-exports; try/except ImportError dual-import boilerplate in every module
- Compression subsystem is heavyweight for its value: a ~2GB XLM-RoBERTa LLMLingua-2 model loaded inside the MCP server, four layered classes where JIT supersedes Hierarchical's role, only two real call sites — while the briefing achieves token economy with simple first-line truncation
- Stop-hook regex decision extraction produces low-quality auto-memories ('Auto-captured from conversation') — memory pollution acknowledged by the existence of a purge_dream_spam maintenance action
- Per-hook cold start: pre_edit/pre_bash boot full SQLAlchemy+aiosqlite per tool call (hence 10s timeouts); the fast raw-sqlite3 path exists in session_start but is dead code
- Recency-only briefing selection — no importance/decay scoring, so trivial recent items displace critical old warnings
- Per-project DB + federation complexity: N SQLite engines opened per briefing for linked projects, path-based links break on moves, an entire consolidation tool needed for restructuring, and ~500 lines of RWLock/LRU/refcount context management exist solely to juggle per-project engines
- Dead speculative modules (tool_search.py, prompt_templates.py) — paper-inspired code never wired in
- ~70 flat config knobs (15 for dreaming alone) plus a second env-invisible CompressionConfig dataclass
- Bootstrap TODO scan stored as category='warning' pollutes the warnings channel the briefing treats as high-signal
- post_edit significance heuristic (keywords like 'auth', 'api', 'config') fires on nearly every real edit, training the model to ignore the nudge

## Reusable Ideas

- The enforcement triad as the core UX: (1) session-start hook injects 'call briefing first' instruction, (2) pre-action hook hard-blocks until a fresh preflight/recall token exists (TTL ~5 min) with the corrective call in the block message, (3) post-action/stop hooks nudge write-back and outcome recording — this is what converts opt-in memory into automatic memory
- Preflight token pattern: consult(preflight) both returns warnings/must_nots AND mints a TTL'd token that gates subsequent write operations, enforced in two places (client hook + server middleware)
- Hooks print instructions, servers do work: keep hook processes dumb, fast (raw sqlite/HTTP ping, sub-second), and fail-open (exit 0 on any error, watchdog kill), letting the MCP tool own all heavy logic
- Small verb-based tool surface (5-8 tools) with action enums, but designed natively with per-action tagged-union schemas instead of flat param unions and a legacy shim layer
- Auto-bootstrap from existing artifacts (manifests, README, git log, TODO scan, CLAUDE.md) so a fresh scope's first briefing is already useful — for Hermes-Brain, bootstrap the global brain from all repos + user config on first run
- Briefing-as-index: failures first, id + first-line truncated summaries, counts, and explicit drill-down hints ('recall(topic) for details'); never dump full content at session start
- Pre-edit file-scoped recall injection: before any edit, surface 'Failed approaches (avoid repeating)' / 'Must NOT do' for that file — the single highest-value context injection in the system
- Read-across/write-local invariant from federation, reborn as scoped queries (global/project/file) over one global store with project provenance columns; keep upstream/downstream/related as inter-project edges
- Ship-inert convention: every new autonomous behavior gets a shadow/dry-run mode default-on, with observability before actuation
