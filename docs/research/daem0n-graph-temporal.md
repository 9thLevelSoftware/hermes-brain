# Research: daem0n:graph-temporal

## Summary

Daem0n-MCP's knowledge-graph/temporal layer is a GraphRAG-lite implementation: a NetworkX DiGraph rebuilt in-memory from three SQLite tables (ExtractedEntity, MemoryEntityRef, MemoryRelationship), populated by pure-regex entity extraction (no LLM, no tree-sitter), deduplicated by string normalization, clustered with real Leiden (igraph/leidenalg) into persisted communities with extractive (non-LLM by default) summaries, and layered over a genuine bi-temporal versioning model (changed_at = transaction time; valid_from/valid_to = valid time) on a MemoryVersion table. Multi-hop queries (causal chains, related-memory expansion, entity evolution) run on the in-memory graph; a RetrievalRouter classifies queries SIMPLE/MEDIUM/COMPLEX and only the COMPLEX path does graph expansion + community summaries. The architecture (dual storage, bi-temporal schema, vector-first/graph-second retrieval, idle-time 'dreaming' maintenance) is sound and worth keeping conceptually; the weakest links are the extraction quality (regex noise), the contradiction module (naive negation regexes, O(N) embedding recompute, and — critically — never wired into the write path), and several O(N^2)/full-scan hotspots that only survive because per-project databases stay small.

## Subsystems

### KnowledgeGraph (dual-storage graph)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\knowledge_graph.py`

In-memory NetworkX DiGraph lazily loaded from SQLite (source of truth). Node naming: 'entity:{id}' and 'memory:{id}'. Load order: (1) all ExtractedEntity rows become entity nodes with attrs (entity_type, name, qualified_name, project_path, mention_count); (2) all MemoryEntityRef rows create memory nodes + memory->entity edges (edge_type='references', relationship='mentions', context_snippet); (3) all MemoryRelationship rows create memory->memory edges (edge_type='relationship', relationship, description, confidence). ensure_loaded() is a no-op after first load; writers call invalidate_graph_cache() / reload_from_db() to force a full clear+reload. Helper methods: get_memories_for_entity (predecessors), get_entities_for_memory (successors), get_related_memories (nx.bfs_tree with depth_limit, default 2), get_common_entities (set intersection), get_entity_neighborhood (1-hop direct + N-hop related). Exposes trace_chain/get_related/trace_evolution delegating to traversal.py.

**Assessment:** Genuinely good pattern for the scale it serves. Full-reload-on-invalidate is crude (every remember() invalidates, so the next graph query reloads the entire DB) but simple and correct. No incremental updates; fine for thousands of memories, breaks for an agent-wide global store. The clean node-namespace convention and SQLite-as-truth/NetworkX-as-index split is worth keeping; a successor needs incremental node/edge upsert instead of full reload.

### Entity extraction (regex heuristics)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\entity_extractor.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\entity_manager.py`

Pure regex, no LLM, no tree-sitter. Patterns: function = identifier followed by '(' (case-insensitive); class = PascalCase ([A-Z][a-z]+ repeated, or 2+ caps then lowercase); file = path-with-extension (ext 1-4 lowercase chars); module = after 'import'/'from'; variable = snake_case inside backticks/quotes. Concepts: quoted strings 3-49 chars, and SCREAMING_SNAKE terms >3 chars. ~30-word STOP_WORDS list filters false positives; per-text dedup on (type, lowercase name); each hit records a ±25-char context snippet and position. EntityManager.process_memory() runs on every remember() when project_path is given (content + rationale concatenated), resolves each entity via EntityResolver, creates MemoryEntityRef(relationship='mentions') if absent, and increments entity.mention_count for repeat mentions.

**Assessment:** Fast, deterministic, zero-cost — but noisy and code-biased. The 'function' pattern matches any word before parens (including English like 'do(', markdown, prose); 'file' regex catches version strings and domains; concepts from quoted strings capture arbitrary phrases. Only relationship type ever auto-created is 'mentions' — there is no typed relation extraction (no subject-predicate-object triples like real GraphRAG). For a successor: keep the cheap regex tier as a first pass, but add an LLM/structured extraction tier for typed entities and relations; the mention_count-as-salience idea is worth keeping.

### Entity resolution / dedup

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\entity_resolver.py`

Canonical key = 'project_path:entity_type:normalized_name' with an in-memory dict cache preloaded per project. Normalization is type-specific string munging: functions camelCase->snake_case then lowercase; classes/modules/variables lowercase; files path-separator normalize + strip './' + lowercase; concepts strip quotes + lowercase. resolve() checks cache, then DB by lower(name) == normalized, then by qualified_name == normalized, else INSERTs a new ExtractedEntity preserving original name and storing normalized form in qualified_name, mention_count=1. Returns (entity_id, is_new).

**Assessment:** Deterministic and cheap, but purely lexical: 'auth service' vs 'authentication service' vs 'AuthSvc' are three entities. No embedding-based or alias-based merging, no cross-project entity identity (project_path is part of the key — a hard blocker for an agent-wide global brain). The lower(name)==normalized DB check is subtly wrong for functions (snake_case conversion applied to the query but only lowercase to the column) — dedup relies mostly on the qualified_name check. Successor should keep exact-normalized fast path but add a semantic-merge tier (embedding similarity + LLM confirm) and global entity IDs with per-scope mentions.

### Leiden community detection + summarization

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\leiden.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\summarizer.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\communities.py`

Real Leiden via igraph + leidenalg: graph converted to undirected, ig.Graph.from_networkx, la.find_partition with ModularityVertexPartition (or CPM with resolution param), seed=42, n_iterations=-1 (to convergence). CommunityManager.detect_communities_from_graph() runs Leiden over the whole mixed memory+entity graph, splits each community's nodes into memory_ids/entity_ids, drops communities with < min_community_size=2 memories, names each 'Entity1 + Entity2 + Entity3' from top-mention entities, uses top-10 entity names as tags. A legacy path (detect_communities) clusters by union-find on memories sharing >=2 tags, dominant tags = present in >50% of members. save_communities() deletes and re-creates all MemoryCommunity rows for the project, generating a summary per community. Summarization defaults to EXTRACTIVE (no LLM): header + member count + top-5 entities + up to 3 truncated (147-char) memories per category, hard-capped at 500 chars; optional LLM mode with anti-hallucination extractive prompt at temperature 0, falling back to extractive on error. summarize_hierarchy() supports bottom-up multi-level summaries (children's summaries become pseudo-members) but level>0 communities are never actually created anywhere — the hierarchy machinery is dead weight. Rebuild is triggered by the idle-time 'dreaming' CommunityRebuild strategy when communities are stale, not on write.

**Assessment:** Leiden bridge itself is excellent: small, correct, deterministic, and the deferred imports keep igraph optional. Running Leiden over the mixed memory+entity graph (rather than a memory-projection) means communities are effectively 'memories sharing entities', which is reasonable. Weaknesses: extractive summaries are shallow (truncated snippets, not synthesis — the GraphRAG value proposition depends on good summaries); hierarchical summarization is scaffolding without a producer (over-engineered); full delete+recreate of communities loses IDs/stability across rebuilds; leiden_community_id is computed but not persisted on MemoryCommunity.

### Community use in retrieval (RetrievalRouter / global search)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\retrieval_router.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\recall_planner.py`

RetrievalRouter classifies each query with ExemplarQueryClassifier into SIMPLE/MEDIUM/COMPLEX (with shadow mode and a confidence threshold fallback to hybrid). SIMPLE -> Qdrant vector-only; MEDIUM -> hybrid BM25/TF-IDF + vector with RRF fusion; COMPLEX -> _graphrag_search: (1) hybrid seeds, (2) graph expansion from top-5 seeds via find_related_memories up to settings.auto_zoom_graph_expansion_depth, scoring expansions as seed_score * 0.8^depth, dedup, re-sort, cap at top_k, (3) attach up to 5 community {name, summary} dicts as community_context — notably NOT matched to the query, just the first 5 communities for the project (ordered by member_count). RecallPlanner defines per-complexity budgets (simple: 3 communities/5 raw, medium: 5/10, complex: 10/20, thresholds 0.5/0.4/0.3) plus JIT compression tiers (soft 4K / hard 8K / emergency 16K tokens). Every step has try/except fallback to hybrid results.

**Assessment:** The routing skeleton and graceful-degradation design are good. But the 'global search' is a stub: community summaries are not ranked against the query (no embedding of summaries, no map-reduce over communities as in real GraphRAG) — top-5-by-size is returned regardless of relevance. Graph expansion with 0.8^depth decay from top-5 seeds is a genuinely useful, cheap idea. Successor: embed community summaries and retrieve them like documents; keep the seed-then-expand pattern.

### Bi-temporal model (MemoryVersion)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\temporal.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\models.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\memory.py`

MemoryVersion table: full snapshot (content, rationale, context, tags, outcome, worked) per version, change_type in {created, content_updated, outcome_recorded, relationship_changed}, changed_at = transaction time (when recorded), valid_from/valid_to = valid time (when true in reality; valid_from NULL means 'same as changed_at' for backwards compat; valid_to NULL means still valid), invalidated_by_version_id FK for supersession chains, composite index on (memory_id, version_number). remember() creates version 1 with valid_from = happened_at (an ISO-8601 tool parameter, tz-naive treated as UTC) — this is the backfill mechanism ('user told me last week X' stored with a past valid_from). get_versions_at_time() is the canonical bitemporal query: (valid_from <= T_valid OR valid_from IS NULL) AND (valid_to IS NULL OR valid_to > T_valid) AND changed_at <= T_transaction. recall(as_of_time=...) deliberately filters by valid_time only (not transaction time) so backfilled facts are findable, calling get_versions_at_time per candidate memory id in a Python loop; used by cognitive/simulate.py to replay decisions at a past time and reflexion/verification for point-in-time claim checks. invalidate_version()/link_memories also append versions (relationship changes create a new version snapshot on BOTH endpoints).

**Assessment:** This is the crown jewel — a correct, simple bi-temporal design (Graphiti-style) with a clear valid/transaction split, append-only audit trail, and a pragmatic default (valid-time-only filtering in recall). Weaknesses: as_of filtering does one query per memory id (N+1); full-snapshot versions are storage-heavy but make point-in-time reads trivial (a fine trade); creating two 'relationship_changed' snapshot versions per link is wasteful noise in the version history; valid_from/valid_to exist only at memory granularity, not on edges (relationships have no temporal validity).

### Contradiction detection

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\contradiction.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\similarity.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\verification.py`

Two parallel mechanisms. (A) graph/contradiction.py: detect_contradictions() embeds new content (ModernBERT), loads ALL currently-valid MemoryVersions (valid_to IS NULL), RE-ENCODES each version's content on the fly (embedding_cache is function-local so it never persists across calls), cosine similarity >= 0.75 triggers a check against ~45 hand-written NEGATION_PATTERNS (regex pairs: not/never/don't/can't... plus antonym pairs like enable/disable, allow/deny, required/optional, works/fails, recommended/deprecated); similarity+negation = Contradiction. invalidate_contradicted_facts() sets valid_to=now and invalidated_by_version_id on the losing version — never deletes (audit trail preserved). CRITICAL FINDING: check_and_invalidate_contradictions is exported from graph/__init__ but NEVER CALLED anywhere in the runtime write path — remember() instead uses (B) _check_conflicts(): hybrid search top-50, similarity >= 0.4/0.5, similarity.detect_conflict() flags 'one worked / one failed' pairs and simple negation-word presence mismatch, and merely attaches a non-blocking 'conflicts' warning to the remember() response. Only has_negation_mismatch() is reused at runtime, by reflexion/verification.py to mark claim-vs-memory conflicts (status='conflict', confidence 0.9).

**Assessment:** The invariant is right (invalidate, never delete; link invalidator to invalidated). The detector is weak: regex negation pairs produce false positives ('use X' vs 'avoid Y' in similar contexts) and miss most real contradictions (semantic opposition without surface negation); the 0.75 threshold is a guess; re-embedding every valid version per call is O(N) encoder passes and would be unusable at scale; and the flagship auto-invalidation path is dead code — in practice contradictions only surface as advisory warnings. Successor: keep the bi-temporal invalidation machinery, replace detection with NLI/LLM-judged contradiction over pre-stored embeddings, and actually wire it into the write path (with human/agent confirmation before invalidating).

### Multi-hop traversal / chain queries

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\graph\traversal.py`

Three query primitives over the in-memory DiGraph. (1) trace_causal_chain(start, end, max_depth=5): nx.all_simple_paths with cutoff, each path formatted with per-edge {relationship, description, confidence}, sorted shortest-first ('how did decision X lead to Y'). (2) find_related_memories(memory_id, relationship_types filter, direction in/out/both, max_depth=2): recursive DFS over memory: nodes only, visited-set dedup, results grouped by relationship type with depth and confidence per hit — also the expansion engine for GraphRAG retrieval. (3) trace_knowledge_evolution(entity_id): collects all memories referencing the entity (graph predecessors), fetches them from DB ordered by created_at, builds a timeline plus a supersession chain from 'supersedes' edges. get_graph_metrics(): node/edge counts by type, density, connected components (undirected), relationship-type histogram. Exposed as MCP tools via tools/graph_tools.py.

**Assessment:** Clean and adequate. all_simple_paths is exponential worst-case — max_depth=5 on a dense graph could blow up (no path-count cap, no timeout). Note paths can route THROUGH entity nodes (memory->entity<-memory is not a directed path so in practice this rarely triggers, but memory->memory chains mixed with 'references' edges are traversed uniformly). The DFS in find_related_memories treats depth per-branch with a shared visited set, so nodes get attributed the depth of whichever branch reached them first, not the minimum — minor scoring inaccuracy in GraphRAG expansion. Keep the three query shapes; add caps.

### Memory-to-memory links (MemoryRelationship)

Files: `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\memory.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\models.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\dreaming\strategies.py`, `C:\Users\dasbl\Daem0n-MCP\daem0nmcp\reflexion\consolidation.py`

Typed directed edges with 5 validated types (VALID_RELATIONSHIPS frozenset in memory.py:42): led_to, supersedes, depends_on, conflicts_with, related_to; plus description and confidence float (default 1.0). Created by: (1) explicit link_memories/unlink_memories MCP tools (validates type, rejects self-links, dedups on (source,target,type), and appends a 'relationship_changed' MemoryVersion snapshot to both endpoints); (2) session compaction — summary memory gets 'supersedes' edges to each archived original; (3) idle-time ConnectionDiscovery dream strategy — scans MemoryEntityRef in a 168h lookback, links pairs sharing >= 2 entities with related_to at confidence 0.7 (max 20 per session, yields to user activity); (4) reflexion consolidation adds supersedes edges when available. Used in recall via the graph: BFS expansion in GraphRAG retrieval, trace_chain, get_related (filterable by type/direction), and supersession chains in evolution tracing. NOTE: links.py is a different thing entirely — ProjectLink rows for cross-project read federation and one-shot DB consolidation (merge linked projects' memories into a target with _merged_from provenance).

**Assessment:** The small closed vocabulary of 5 edge types is a strength — legible, enforceable, meaningful to an LLM. Auto-linking via shared entities during idle time is a genuinely good continual-learning idea (cheap, incremental, interruptible). Weaknesses: 'confidence can decay over time' is documented but no decay is implemented; conflicts_with edges are never auto-created by the contradiction detector (the two systems don't talk); versioning both endpoints on every link bloats version history. ProjectLink federation (read-many/write-own) is the right isolation model but its consolidation path duplicates memories rather than referencing them.

## What Worked

- Dual-storage pattern: SQLite as durable source of truth + NetworkX in-memory index for traversal — simple, debuggable, no graph-DB dependency
- Bi-temporal MemoryVersion schema (changed_at vs valid_from/valid_to, invalidated_by_version_id, append-only invalidation) — correct design, cleanly implemented, actually used by recall(as_of_time), decision simulation, and claim verification
- happened_at backfill parameter flowing from MCP tool -> remember() -> version.valid_from, with the pragmatic choice to filter recall by valid-time only so backfilled facts are discoverable
- Real Leiden (igraph/leidenalg, seed=42, modularity) instead of a homegrown clustering — small correct bridge module, deterministic output
- Vector-first / graph-second retrieval: hybrid seeds then graph expansion with seed_score * 0.8^depth decay, every stage wrapped in fallback-to-hybrid try/except
- Closed vocabulary of 5 typed memory edges (led_to, supersedes, depends_on, conflicts_with, related_to) validated at the API boundary
- Idle-time 'dreaming' maintenance: ConnectionDiscovery auto-links memories sharing >=2 entities (confidence 0.7, capped, interruptible), CommunityRebuild refreshes communities only when stale
- Query complexity routing (SIMPLE/MEDIUM/COMPLEX) with shadow mode and confidence-threshold fallback — safe rollout mechanics for a new retrieval path
- Entity mention_count as a cheap salience signal for naming communities and ranking entities

## Weaknesses

- Contradiction auto-invalidation (check_and_invalidate_contradictions) is dead code — never called in any write path; remember() only attaches advisory conflict warnings, so the bi-temporal invalidation machinery is largely unexercised in production
- Contradiction detector re-embeds EVERY currently-valid version on each call (function-local cache), O(N) encoder passes; negation-regex pairs are both noisy and blind to non-surface contradictions
- Entity extraction is regex-only and noisy (any word before '(' is a 'function'; quoted prose becomes 'concepts'); only 'mentions' relations, no typed subject-predicate-object extraction
- Entity resolution is purely lexical and project-scoped — no semantic merging of aliases, and project_path baked into the canonical key blocks a global agent-wide entity space
- Community 'global search' is a stub: top-5 communities by member size attached as context regardless of query relevance; summaries are truncated snippet concatenations by default, not synthesis; hierarchical (level>0) summarization code has no producer
- Full graph reload on every invalidation, full delete+recreate of communities on rebuild (unstable community IDs), N+1 per-memory version queries in as_of recall, O(N^2) pairwise loops in tag clustering — all fine per-project, all fatal for a global store
- nx.all_simple_paths with cutoff=5 has no path-count cap or timeout; DFS expansion attributes first-seen depth rather than minimum depth
- Edges have no temporal validity (no valid_from/valid_to on relationships) and the documented confidence decay is unimplemented; contradiction detection never creates conflicts_with edges

## Reusable Ideas

- Carry the bi-temporal MemoryVersion schema forward nearly as-is: changed_at (transaction) + valid_from/valid_to (valid) + invalidated_by_version_id, append-only invalidation, happened_at backfill parameter, and valid-time-only default filtering in recall — but add temporal validity to edges too, and make as_of filtering a single SQL join instead of per-memory queries
- Keep SQLite/Postgres-as-truth + in-memory graph index, but with incremental upsert of nodes/edges on write instead of invalidate-and-full-reload
- Two-tier entity pipeline: keep the free regex/normalization tier as a synchronous first pass, add an async LLM tier for typed entities and typed relations (which would finally auto-populate led_to/depends_on/conflicts_with edges); global entity IDs with per-project/per-scope mention tables instead of project-scoped entities
- Wire contradiction handling into the write path for real: detect via stored embeddings + NLI/LLM judgment (not negation regexes), create a conflicts_with edge, and invalidate the losing version via the existing bi-temporal mechanism — optionally gated on agent confirmation
- Keep Leiden (seeded, modularity) for communities but: persist stable community identity across rebuilds (match by member overlap), embed community summaries and retrieve them like documents for true global search, and generate summaries with an LLM using the existing anti-hallucination extractive-prompt pattern (temp 0, source-ID attribution, 'only what is stated')
- Keep the retrieval router shape: complexity classification with shadow mode + confidence fallback, hybrid seeds -> graph expansion with geometric depth decay (0.8^depth) -> community context, JIT compression tiers (4K/8K/16K)
- Keep idle-time dreaming strategies as the home for graph maintenance: shared-entity auto-linking (>=2 shared entities, capped, interruptible, sub-1.0 confidence) and staleness-gated community rebuilds
- Keep the 5-type closed edge vocabulary and the read-linked/write-own project federation model (ProjectLink) as the multi-scope story, replacing copy-based consolidation with reference-based federation
